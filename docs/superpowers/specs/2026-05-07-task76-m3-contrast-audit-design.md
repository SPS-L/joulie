# TASK-76 — M3 Token Contrast Audit (Design Spec)

**Filed:** 2026-05-07
**Backlog row:** TASK-76, 🟡, depends on TASK-18 (done)
**Goal:** Verify every WCAG-relevant text/surface pair in the Joulie M3 ramps meets WCAG 2.1 AA (≥ 4.5:1 normal text, ≥ 3:1 large text + non-text UI), light + dark. Lock the result in with a JVM unit test so any future M3 re-seed that drops a pair below threshold breaks the build.

## 1. Background

The original backlog row flagged `#FB8C00` (DC orange tertiary) + white-on-tertiary text as the candidate to investigate. That ramp was already retired by TASK-57 (Brand Pack v1.0): the tertiary seed is now `#FFD54D` Joulie yellow, with light-tertiary text `#7F5700` on container `#FFE082`. The historical concern is gone, but the audit itself is still worth doing — both ramps were generated by Material Theme Builder from brand seeds, and an automated check that all token pairs clear AA is a much stronger guarantee than "MTB output is AA by construction" trust.

The current state of `app/src/main/res/values/colors.xml` (light) and `app/src/main/res/values-night/colors.xml` (dark) is the source of truth.

## 2. Scope

**In scope (this PR):**

- Every M3 text-on-surface pair in light + dark themes (`primary/onPrimary`, container variants, `surface/onSurface`, `surfaceVariant/onSurfaceVariant`, `inverseSurface/inverseOnSurface`).
- Non-text UI pairs (`outline/surface`, `outlineVariant/surface`) at 3:1.
- Brand-pack additions wired as text or surface colours: `joulie_ink_deep` light + dark (wordmark on background), `joulie_brand_blue` (the white "J" on the launcher tile is `onPrimary`-equivalent contrast).

**Out of scope (filed elsewhere or skipped):**

- Material component states (Button enabled/disabled/pressed, Switch on/off, FAB, Chip, TextField error, Snackbar, BottomNav selected). TASK-77 already covers the ChargeEdit toggle group; broader component-state contrast is a forward-work follow-up.
- Real-screen sweep / dynamic color (S+) variants. TASK-78 covers the real-device TalkBack walkthrough; a screenshot-contrast sweep would be a separate task.
- `chart_1..chart_5` (chart series strokes, not text/UI tokens), `joulie_spark_green` (accent strip, never text — `colors.xml` already documents the ~1.6:1 failure if misused).

## 3. Architecture

| Unit | Responsibility | Path | Public surface |
|------|----------------|------|----------------|
| `ContrastRatio` | Pure WCAG 2.1 §1.4.3 ratio computation. ARGB int → relative luminance → ratio. No Android dep. | `app/src/test/java/org/spsl/evtracker/testing/ContrastRatio.kt` | `fun ratio(fg: Int, bg: Int): Double` |
| `M3ContrastAuditTest` | Parameterised JUnit test asserting each audited pair clears its threshold. Fails with `expected ≥ 4.5:1, got 3.91:1 for (primary=#1F3DCC, onPrimary=#FFFFFF)`. | `app/src/test/java/org/spsl/evtracker/M3ContrastAuditTest.kt` | One JUnit class, no public API |
| `docs/DESIGN.md §11.2` | Documents the audit: what's covered, thresholds, source-of-truth pointer to the test. | DESIGN.md | Documentation |

The helper lives in `testing/` (test-source-set), not in `app/src/main`, because no runtime code path needs it. Keeping it test-only is YAGNI-correct and avoids growing the production APK by a few hundred bytes for code only the test consumes.

The test hard-codes hex values copied from `colors.xml`. This is deliberate — it's a contract test. The test exists *because* the XML values must not regress; reading them back from XML at runtime would let the test silently pass while the regression slipped through. A code comment in the test points to the canonical XML files.

## 4. Audit matrix

WCAG 2.1 AA thresholds:

- **Normal text:** ≥ 4.5:1 (WCAG 1.4.3). Anything an M3 `onX/X` pair would back as body / label text.
- **Large text + non-text UI:** ≥ 3:1 (WCAG 1.4.3 + 1.4.11). `outline*` strokes, focus rings, large-text headlines.

## 5. WCAG ratio formula

WCAG 2.1 §1.4.3 defines:

```
sRGB → linear:  c_lin = c_srgb / 12.92                   if c_srgb ≤ 0.03928
                       ((c_srgb + 0.055) / 1.055) ^ 2.4   otherwise
relative L   :  L = 0.2126 R_lin + 0.7152 G_lin + 0.0722 B_lin
ratio        :  (L_lighter + 0.05) / (L_darker + 0.05)
```

`c_srgb` are channel values in [0, 1]. The helper reads ARGB ints (`Color.parseColor("#RRGGBB")`-equivalent), discards alpha, splits into RGB, and runs the formula. Returns Double.

## 6. Test design

```kotlin
class M3ContrastAuditTest {
    @Test fun lightTheme_allPairsClearThreshold() {
        LIGHT_PAIRS.forEach { pair ->
            val ratio = ContrastRatio.ratio(pair.fg, pair.bg)
            assertTrue(
                "${pair.name}: expected ≥ ${pair.threshold}:1, got %.2f:1 for (fg=#%06X, bg=#%06X)"
                    .format(ratio, pair.fg and 0xFFFFFF, pair.bg and 0xFFFFFF),
                ratio >= pair.threshold,
            )
        }
    }

    @Test fun darkTheme_allPairsClearThreshold() { /* mirror */ }
}
```

The pair list lives in companion-object `val`s; each entry is a tiny data class `Pair(name, fg, bg, threshold)`. No reflection, no parameterised-runner gymnastics — JUnit 4's `forEach` + per-pair `assertTrue` is the simplest path that still produces a precise failure message naming the offending pair.

If multiple pairs fail in the same run, `forEach` with `assertTrue` short-circuits on the first. This is acceptable: failure → fix → re-run is the workflow, and surfacing all failures at once isn't worth the complexity. (Easy to upgrade later by collecting failures into a list and asserting empty if needed.)

## 7. TDD plan

The implementation plan (`docs/superpowers/plans/2026-05-07-task76-m3-contrast-audit.md`) decomposes this into four RED-GREEN cycles:

1. `ContrastRatio.ratio` against a single known reference pair (e.g. `(white, black) = 21.0`).
2. `ContrastRatio.ratio` for `(white, blue #1F3DCC)` against an externally-computed reference value.
3. `M3ContrastAuditTest.lightTheme_allPairsClearThreshold` with the full light matrix.
4. `M3ContrastAuditTest.darkTheme_allPairsClearThreshold` with the full dark matrix.

Each step writes a failing test first, runs it to verify it fails for the expected reason, then writes the minimum code to pass.

## 8. Acceptance criteria

1. **`./gradlew :app:testDebugUnitTest --tests "org.spsl.evtracker.M3ContrastAuditTest"` passes locally and in CI.**
2. **Mutation kill verified:** before merge, deliberately corrupt one token in the audit matrix (e.g. flip light `onSurfaceVariant` from `#43474E` to a too-light `#A0A0A0`), run the test, observe a failure with a precise message naming the pair and the actual ratio, then revert. Record the observed ratio in the release commit.
3. **`docs/DESIGN.md §11.2`** documents the audit: scope, thresholds, audited-pair count, source-of-truth pointer.
4. **`docs/BACKLOG.md`** flips TASK-76 to ☑ Done with an outcome banner naming the worst-case ratio observed (lowest passing pair) so future contributors know how much headroom we have.
5. **No production-source-set churn.** All new code lives in `app/src/test/`. `colors.xml` is unchanged unless an audit failure forces a re-seed (in which case the failure is the news, the re-seed is the fix, both go in the same PR).

## 9. Risks + mitigations

| Risk | Mitigation |
|------|------------|
| MTB-derived tokens already pass AA, so the test is "always green" and adds no value. | The lock-in *is* the value. Future M3 re-seeds (e.g. when AGP / MTB versions change, or when brand seeds shift) need a deterministic guard. The test is exactly that guard. |
| Hard-coded hex drifts away from `colors.xml`. | The contract test failing on drift is the signal we want — but to catch *intentional* drift fast, the DESIGN.md §11.2 entry names both files (XML + test) as paired source-of-truth. A CI lint that diffs them is overkill (TASK-79+ if it ever bites). |
| WCAG 2.1 vs WCAG 2.2 / APCA divergence. | This audit anchors on WCAG 2.1 AA (the legally cited standard for accessibility regulations in EU + US). APCA is still draft; no need to track it now. |

## 10. Out of scope (future)

- Component-state contrast (filled-button disabled, switch off, error-state text-field outline). TASK-77 covers one slice; broader sweep is forward-work.
- Dynamic color (Material You S+) variants. The user's wallpaper-derived ramp can fall below AA; mitigation is to pin our MTB-seeded ramp as the override on screens where contrast matters, but this is a much larger UX call.
- Screenshot-contrast sweep across real screens. Would need Roborazzi (TASK-35) to land first.
